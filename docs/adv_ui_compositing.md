# Compositing Adv-mode UI on top of a background

How to reproduce the in-game ADV-mode dialog box (NormalPrinter + NamePlateBase) over a scene background, matching the look of an actual game screenshot.

A working reference implementation is checked in alongside this document — see *Reference script* at the bottom.

## Render order (back to front)

1. **Background** — full-canvas scene image, scaled to *cover* a 2560×1440 canvas (center-crop horizontally; backgrounds in this game are 4096×2048, i.e. 2:1 vs the 16:9 canvas, leaving 320 px of pan headroom on each side that we discard).
2. **Characters** — layered character composites from `characters/<Name>/`, each emitted by `extract_bundle.py` with its own `pos`-based schema. Character compositing is out of scope for this document; see the existing renderer in `app.js`.
3. **NormalPrinter UI** — the bottom dialog frame (`Frame_Top`, `Frame_Bottom`, `Screen`, `Grass_1`, `Grass_2`) and the floating speaker plate (`NamePlateBase`). All six sprites are extracted as PNGs into `ui/UI_Adv/` by `extract_ui_layers.py`.

## Canvas conventions

- Canvas size: **2560×1440** (the game's reference resolution; matches every NormalPrinter rect that anchors to canvas extremes).
- Unity uses **Y-up** with origin at the bottom-left; PIL uses **Y-down** with origin at the top-left. Convert at the end: `pil_top = canvas_h - (unity_bottom + height)`.
- Coordinates and sizes from the prefab are in canvas pixels (the parent Canvas is in Screen Space — Overlay at reference resolution, so RectTransform numbers map 1:1 to canvas pixels).

## RectTransform → canvas rect

For each child rect with parameters (`anchor_min`, `anchor_max`, `anchored_position`, `size_delta`, `pivot`) and parent rect `(left, bottom, w, h)` in canvas Unity coords:

```
anchor_left   = parent.left   + parent.w * anchor_min.x
anchor_right  = parent.left   + parent.w * anchor_max.x
anchor_bottom = parent.bottom + parent.h * anchor_min.y
anchor_top    = parent.bottom + parent.h * anchor_max.y

child.w = (anchor_right - anchor_left) + size_delta.x
child.h = (anchor_top   - anchor_bottom) + size_delta.y

pivot_x = (anchor_left   + anchor_right) / 2 + anchored_position.x
pivot_y = (anchor_bottom + anchor_top)   / 2 + anchored_position.y

child.left   = pivot_x - pivot.x * child.w
child.bottom = pivot_y - pivot.y * child.h
```

Two cases the formula collapses cleanly:
- `anchor_min == anchor_max` → `child_size = size_delta`, position offset by `anchored_position` from the anchor point.
- `anchor_min == (0,0)`, `anchor_max == (1,1)`, `size_delta == (0,0)` → child stretches to fill the parent (a "passthrough container").

To convert to PIL paste coordinates: `pil_xy = (round(child.left), round(canvas_h - (child.bottom + child.h)))`.

## Where the layout lives

The authoritative source is `ui/NormalPrinter.json`, generated by `scripts/extract_ui_layers.py` from `naninovel-textprinters_assets_all.bundle` + `general-sprites_assets_all.bundle`. The file carries each leaf's resolved canvas position, sprite reference (as `(atlas, sprite_name)` plus a relative `file` path into `ui/<atlas>/`), `m_Color` override, `m_Material`, and any `CanvasGroup` gating that affected the walk. Prefer driving compositors from that file rather than hardcoding sprite lists; the table below is a derived snapshot of what the JSON contains for NormalPrinter, kept for at-a-glance reference.

**Convention for prefab roots**: the root node's RectTransform is typically a stub (`anchors=(0,0)..(0,0), size=(0,0)`), because at runtime the prefab is parented under a Canvas that forces it to canvas-size. The compositor must treat the root's effective rect as the canvas, regardless of its serialized values, and apply the rect formula to its children. (See `is_root` handling in the reference compositor below.)

## Background scaling

```
scale = max(canvas_w / bg_w, canvas_h / bg_h)
bg.resize((bg_w*scale, bg_h*scale), LANCZOS)
center-crop to (canvas_w, canvas_h)
```

For the 4096×2048 backgrounds in `backgrounds/main/`: scale ≈ 0.703, intermediate size ≈ 2880×1440, then crop 160 px off each side. The discarded margins are camera pan headroom — Naninovel's `@camera` pans within them at runtime; freezing the camera at center is the right default.

## Compositing in linear space

This step is necessary, not optional — straight-sRGB-byte compositing (PIL's `alpha_composite`) makes the dialog band visibly darker than the in-game look.

Why: Unity's UI pipeline in **Linear color space** (the default for modern Built-in / URP / HDRP projects) samples sRGB-tagged textures into linear, blends linearly, then writes back to sRGB at present. PIL blends sRGB bytes directly. For the `Screen` sprite (RGB=(0,0,0), mean alpha ≈ 35%) over a mid-gray destination, the difference is roughly `83/255` (sRGB-straight) vs `~111/255` (linear-correct) — a 34% apparent-brightness lift on every dst pixel that participates in a low-alpha blend. The bottom dialog band stacks `Screen` + `Frame_Bottom` + `Grass_2` over the same background region, so the cumulative darkening in the naive pipeline is exactly the "too dark" effect.

Algorithm (per RGBA pixel, RGB in `[0,1]`, α in `[0,1]`):

```
src_lin = srgb_to_linear(src.rgb)
dst_lin = srgb_to_linear(dst.rgb)
out_lin = src_lin * src.a + dst_lin * (1 - src.a)
out_a   = src.a + dst.a * (1 - src.a)
out.rgb = linear_to_srgb(out_lin)
```

Standard sRGB ↔ linear curves (do not use a 2.2 power approximation):

```
srgb_to_linear(c) = c/12.92                 if c <= 0.04045
                  = ((c+0.055)/1.055)^2.4   otherwise

linear_to_srgb(c) = c*12.92                 if c <= 0.0031308
                  = 1.055 * c^(1/2.4) - 0.055   otherwise
```

Alpha is **not** gamma-corrected — it's a coverage value.

Where this matters most:
- `Screen` (alpha ~35%, RGB=0): biggest single delta — most dst-weight in the blend.
- `NamePlateBase` (alpha ~77%, RGB=(63,42,45)): visible saturation/lightness lift over bright background regions.
- `Frame_Top`, `Frame_Bottom`, `Grass_*` (alpha ≥ 94%): nearly invisible delta — the blend is dominated by src, so color-space differences collapse.

## Things this composite does *not* model

- **`MessageLabel` / `AuthorLabel` text** — these are TextMeshPro at runtime; they don't exist as sprites. Add them via your text rendering layer.
- **`InputIndicator` / `SkipModeIndicator`** — both placed but not included in the composite above; see the original prefab if you need them.
- **`Stream` mode** — the sibling `Stream/Frame` (`StreamFrame` 2560×1440) is gated by a `CanvasGroup` with `m_Alpha = 0` in the prefab — it's the alternate full-screen "stream" presentation that's hidden by default. Skip it for the standard dialog look.
- **Camera post-processing** — color grading, bloom, exposure compensation. These aren't in any AssetBundle (they live in Unity PlayerSettings / scene Volume components). If the linear-space composite still looks slightly off vs an in-game screenshot, this is the residual gap.
- **Character placement between background and UI** — the renderer in `app.js` already handles layered characters via the `characters/<Name>/layers.json` schema; integrate by drawing characters between steps 1 and 3 of the render order.
- **CanvasScaler dynamic resizing** — at native 2560×1440 every number above is in pixels. If you target a different resolution, multiply all positions and sizes by the same scale factor.

## Reference compositor

`scripts/compose_ui_panel.py` drives directly from `ui/NormalPrinter.json` (and any other prefab JSON the extractor produced — 39 prefabs in `ui/meta.json`). The same compositor handles screen prefabs (`NormalPrinter`, `OptionsUI`, `TitleUI`, …) and panel prefabs with runtime-instantiated widgets (`TrialChoicePanel_*` + `ChoiceButton_Trial_*`) — see [`trial_ui_compositing.md`](./trial_ui_compositing.md).

```bash
# Render NormalPrinter chrome (no widgets) over a background.
python3 scripts/compose_ui_panel.py NormalPrinter - /tmp/normal_printer.png backgrounds/main/Background_002_001.png
```

The compositor:
1. Cover-fits the background (or uses solid black if omitted).
2. Renders the panel's static `layers` in z-order, resizing each sprite to `layer.size` for any genuinely-stretched Unity Image renders (a no-op when the sprite pack flush).
3. For each `containers[i]` with a `placement`, walks the supplied widget list through the layout-group formula and composites each widget's layers at the resolved offset.

Linear-space alpha compositing throughout; canvas saved back to sRGB on output.
